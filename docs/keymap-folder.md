# nvWgW/ — the keymap folder

This folder is the **source of truth** for your Moonlander layout. It's
generated by ZSA's Oryx web configurator and then merged with your
custom QMK code via the GitHub Action or local `build.sh`.

## Files

| File | Purpose |
|---|---|
| `keymap.c` | C source for the keymap. Defines layers, custom keycodes, DUAL_FUNC handlers, RGB, etc. |
| `config.h` | C header for keyboard config. USB settings, serial number, etc. |
| `keymap.json` | ZSA marker file. Tells QMK to load the `zsa/oryx` module. Don't edit. |
| `rules.mk` | Feature flags. Tells QMK which features to compile in (autocorrect, tap dance, etc.). |

## Editing

**For most changes** (key positions, layer mappings, lighting), edit in
Oryx and let the GitHub Action sync it. Don't edit `keymap.c` directly
for layout changes — your edits will be overwritten by the next Oryx
sync.

**For QMK-specific changes** (DUAL_FUNC tweaks, new custom keycodes,
new RGB schemes), edit `keymap.c` directly. These changes survive the
Oryx merge because the action uses `git merge -Xignore-all-space oryx`,
which preserves content not present in the Oryx output.

**For feature flags** (adding a new `*_ENABLE = yes`), edit `rules.mk`.

## Layer table

The `keymaps[][MATRIX_ROWS][MATRIX_COLS]` array in `keymap.c` is a 2D
array: layers × matrix positions. Each cell is a keycode.

The Moonlander has 6 rows × 17 columns = 72 keys per layer. The
`LAYOUT_moonlander` macro arranges the keys into the physical Moonlander
shape.

## Custom keycode pattern

To add a new custom behavior (e.g. "tap to type `;`, hold to type `:`"):

1. Add a `#define` at the top of `keymap.c`:
   ```c
   #define MY_SEMI LT(15, KC_SCLN)
   ```
2. Use it in a keymap position:
   ```c
   [0] = LAYOUT_moonlander( ..., MY_SEMI, ... )
   ```
3. Add a case to `process_record_user()` to handle tap vs hold:
   ```c
   case MY_SEMI:
     if (record->tap.count > 0) {
       // tap behavior
       if (record->event.pressed) register_code16(KC_SCLN);
       else unregister_code16(KC_SCLN);
     } else {
       // hold behavior
       if (record->event.pressed) register_code16(LSFT(KC_SCLN));
       else unregister_code16(LSFT(KC_SCLN));
     }
     return false;
   ```
4. Build with `./build.sh`

## Why so much code for one tap/hold behavior?

Because QMK's built-in `MT(mod, kc)` only supports ONE modifier on hold
and ONE key on tap. To do "tap = letter, hold = shifted symbol" (which
requires holding a modifier AND pressing a key) you need custom code.

ZSA's Oryx tool generates this code for you automatically. The
DUAL_FUNC_* macros are the result.
